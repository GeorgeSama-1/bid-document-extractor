# GPU 文档提取 Web 服务设计

## 目标

将现有 `pdf-toc-pipeline` CLI 包装为运行在内网 GPU 服务器上的单用户 Web 服务。用户从本地浏览器访问 `http://172.20.0.160:<port>`，上传一个 PDF，为每个任务选择物理 GPU，并填写 VLM Endpoint、模型名和 API Key。服务异步执行任务，展示进度与错误，任务结束后支持浏览结果、下载单个文件和下载完整 ZIP。

服务部署在可信内网，不提供登录功能。服务监听 `0.0.0.0`，网络访问控制由现有内网和服务器防火墙承担。

## 已确认的范围

- 一个任务只处理一个由浏览器上传的 PDF。
- GPU 由用户从服务器实际可用 GPU 列表中选择。
- 第一版所有任务都启用 PP-Structure，因此 GPU 是必选项；不提供纯 CPU/关闭 PP-Structure 的任务模式。
- 同一 GPU 的任务按提交顺序排队；不同 GPU 的任务可以并行。
- 每个任务可以独立设置 VLM Endpoint、模型名、API Key、超时、最大 token 数和并发数。
- API Key 不写入 SQLite、日志、输出文件或下载包；任务结束或取消后从服务内存移除。
- 支持任务列表、任务详情、阶段进度、日志尾部、取消任务、结果浏览、单文件下载和完整 ZIP 下载。
- 复用现有结果浏览页面和 API 能力，不改变 `pdf-toc-pipeline` 的文档处理语义。
- 第一版不提供账号、权限系统、批量 PDF、跨服务器调度、Redis/Celery 或任务优先级。

## 方案选择

采用“FastAPI + SQLite + 独立子进程 + 每卡 FIFO 队列”。FastAPI 负责 HTTP 与静态页面；SQLite 持久化非敏感任务状态；调度器为每张 GPU 串行运行子进程；子进程继续调用现有 CLI。

不采用进程内直接调用流水线，因为并行任务无法安全地共享和修改 `CUDA_VISIBLE_DEVICES`，任务取消和故障隔离也较差。不采用 Redis/Celery，因为单机、单用户场景不需要额外服务依赖。

## 组件边界

### Web API

职责是校验上传和表单、调用任务服务、返回稳定的 JSON 或文件响应。API 不直接启动子进程，也不持有队列实现细节。

接口：

- `GET /api/system/gpus`：返回 `nvidia-smi` 检测到的物理 GPU 编号、名称、总显存和当前显存占用。
- `POST /api/jobs`：以 multipart 表单上传 PDF 和任务参数，创建任务。
- `GET /api/jobs`：按创建时间倒序返回任务列表。
- `GET /api/jobs/{job_id}`：返回状态、排队位置、阶段进度、脱敏参数、错误和最近日志。
- `POST /api/jobs/{job_id}/cancel`：取消排队任务，或终止运行中任务的进程组。
- `GET /api/jobs/{job_id}/files`：返回输出目录下允许下载的文件树。
- `GET /api/jobs/{job_id}/files/{file_path:path}`：下载任意层级的单个输出文件。
- `GET /api/jobs/{job_id}/archive`：按需生成并下载该任务完整输出 ZIP。

文件树、单文件下载和 ZIP 枚举使用同一个安全遍历器。遍历器拒绝路径穿越，完全跳过符号链接，并验证每个解析后的文件仍位于该任务输出根目录。任何指向根目录外或无法安全解析的项目都不出现在列表或 ZIP 中，也不能下载。

### 任务服务

职责是组合任务存储、内存密钥仓、GPU 清单和调度器，为 API 提供单一业务接口。它负责生成不可预测的 UUID 任务 ID、保存上传文件、创建独立输出目录，以及在任务完成后清除密钥。

任务目录约定：

- 上传：`service_data/uploads/<job_id>/input.pdf`
- 输出：`outputs/job_<job_id>/`
- 数据库：`service_data/jobs.sqlite3`
- 运行日志：`service_data/logs/<job_id>.log`
- ZIP：按需生成到 `service_data/archives/<job_id>.zip`

上传文件名只作为展示字段保存；服务器路径不使用客户端文件名。上传必须具有 PDF 扩展名和 `%PDF` 文件头，并受可配置的最大上传大小限制。

任务成功后，任务详情返回稳定字段 `run_name="job_<job_id>"`。现有结果浏览接口继续接收这个 `run_name`，即映射到 `outputs/job_<job_id>/`。只有 `succeeded` 任务展示“浏览材料”入口；页面用该字段调用现有 `/api/runs/{run_name}/...` 接口。

### SQLite 任务存储

保存任务 ID、原始文件名、选定 GPU、非敏感 VLM 配置、流水线开关、状态、阶段进度、时间戳、PID、退出码、输出路径和错误摘要。

状态集合为 `queued`、`running`、`succeeded`、`failed`、`cancelled`。状态转换由任务服务集中控制。API Key 永不进入任务模型的持久化表示。

服务启动时将遗留的 `running` 任务标记为 `failed`，原因是服务重启导致子进程状态不可恢复。由于 Key 不持久化，遗留的 `queued` 任务也标记为 `failed`，并提示重新提交。

排队位置只统计相同 GPU 队列中排在当前任务之前的 `queued` 任务，首个等待任务为 1；`running` 和终态任务返回 `null`。

### 内存密钥仓

只保存 `job_id -> API Key`，以锁保护。任务进入子进程前读取，任务成功、失败、取消或服务关闭时删除。任务详情和普通日志格式化逻辑没有读取 Key 的接口。

所有来自子进程、依赖异常和请求校验的文本，在进入日志、SQLite、API 错误响应或任务详情前，必须经过该任务的精确脱敏器：将完整 Key 替换为固定文本 `[REDACTED]`。任务创建接口手工提取并校验 multipart 字段，Key 不进入可被 FastAPI/Pydantic 默认 422 响应回显的持久化请求模型。测试必须用唯一哨兵 Key 验证数据库、日志、API 响应和 ZIP 全部不包含该值。

### GPU 清单

通过 `nvidia-smi --query-gpu` 获取物理 GPU。创建任务时验证选择的编号仍然存在。第一版任务始终启用 PP-Structure 并必须选择 GPU；若 `nvidia-smi` 不可用或没有 GPU，GPU 列表接口返回明确错误并禁止创建任务。

GPU 清单实现通过接口注入，测试使用假实现，不依赖测试机显卡。

### 每卡调度器

每个物理 GPU 对应一个 FIFO 队列和一个工作线程。一个工作线程同一时间只运行该 GPU 的一个子进程；不同 GPU 的线程互不阻塞。

该架构强制服务以单实例、单 Python 进程、单 Uvicorn worker 运行。启动时获取 `service_data/service.lock` 的排他文件锁，第二个实例无法启动。systemd 的 `ExecStart` 固定使用 `--workers 1`，禁止以多 worker 模式部署，否则进程内队列和密钥仓无法保证一致性。

调度器启动子进程时：

- 使用当前 Python 解释器执行 `-m bid_knowledge.cli pdf-toc-pipeline`。
- 设置 `CUDA_VISIBLE_DEVICES=<物理GPU编号>`。
- CLI 参数使用 `--pp-structure-device gpu`，因此子进程只使用映射后的可见 GPU。
- VLM Endpoint 和模型名作为普通 CLI 参数传递。
- API Key 只放入子进程环境变量，并通过 `--vlm-table-api-key-env VLM_API_KEY` 指定变量名，不出现在命令行。
- 为子进程创建独立进程组，以便取消时先发送温和终止信号，超时后强制结束整个进程组。
- 合并读取 stdout/stderr，写入任务日志并解析 `[当前步/总步数]` 行，更新阶段进度。

运行时参数采用以下明确默认值和范围：

- 最大上传大小：默认 500 MiB，由 `BID_SERVICE_MAX_UPLOAD_BYTES` 覆盖，必须为正整数。
- `path_root`：默认 `PDF`，长度 1–100。
- PP-Structure 固定启用；三个预处理开关默认均为 `false`。
- VLM 默认启用；Endpoint 必须是无内嵌用户名/密码的绝对 `http` 或 `https` URL；模型名长度 1–200；API Key 可以为空。
- VLM 超时：默认 1800 秒，范围 1–3600。
- VLM 最大 token 数：默认 8192，范围 256–32768。
- VLM 并发数：默认 16，范围 1–128；上限可由 `BID_SERVICE_MAX_VLM_WORKERS` 下调但不能高于 128。

页面允许覆盖 `path_root`、PP-Structure 三个预处理开关，以及全部 VLM 字段。创建任务时完成所有范围校验，避免任务排队后才失败。

服务关闭时先停止接收新任务并取消所有排队任务，再向每个运行任务的进程组发送终止信号，最多等待 30 秒，随后强制结束仍存活的进程组。只有确认进程已退出后才写入 `cancelled` 或 `failed` 终态并清除 Key。systemd 使用 `KillMode=control-group` 和至少 45 秒的 `TimeoutStopSec`，确保异常停止时旧子进程不会脱离服务继续占用 GPU。启动文件锁也防止旧实例尚存时启动新调度器。

## 页面设计

现有静态结果浏览页扩展为三个区域：

1. “新建任务”表单：PDF、GPU、目录根名、三个 PP-Structure 预处理开关、VLM Endpoint、模型、Key，以及折叠的高级参数。PP-Structure 与 VLM 在第一版固定启用，不展示总开关。
2. “任务列表”：显示排队/运行/成功/失败/取消状态、GPU、当前阶段、时间和操作按钮。
3. “任务详情与结果”：轮询任务状态，展示阶段进度、最近日志、结果文件树、单文件下载、完整 ZIP 下载，并复用已有材料结果浏览功能。

Key 输入框使用密码类型，页面不将 Key 写入 localStorage、URL、页面日志或任务详情。提交成功后立即清空 Key 输入值。

页面使用现有原生 HTML/CSS/JavaScript，不增加前端构建工具。任务状态使用短轮询，避免为了单机服务引入 WebSocket 复杂度。

## 数据流

1. 浏览器读取 GPU 列表并填写任务表单。
2. 浏览器以 multipart 上传 PDF 和参数。
3. API 校验输入；任务服务保存 PDF、将非敏感字段写入 SQLite、将 Key 放入内存并将任务加入所选 GPU 队列。
4. 该 GPU 空闲后，调度器把任务置为 `running`，以隔离环境启动 CLI 子进程。
5. 子进程输出驱动日志和阶段进度更新；浏览器轮询详情接口。
6. 子进程退出码为 0 时任务变为 `succeeded`，否则为 `failed`；任何终态都会清除内存 Key。
7. 成功后浏览器读取文件树，安全下载单个文件或请求生成 ZIP。

## 错误处理

- 无效 PDF、GPU、URL 或数值参数：创建任务前返回 4xx，不产生半成品任务。
- 上传中断：清理未完成的临时文件。
- VLM Endpoint 或模型为空：拒绝提交，避免现有流水线静默跳过。
- API Key 可为空，以兼容不需要鉴权的内部 VLM Endpoint。
- 子进程启动失败或非零退出：任务标记失败，保留脱敏日志和已有输出供排查。
- 页面刷新或浏览器关闭：任务继续运行，状态可从 SQLite 恢复查看。
- 服务重启：不可恢复的排队/运行任务标记失败，不自动重放，避免在没有 Key 的情况下产生错误结果。
- 取消：排队任务直接取消；运行任务终止整个进程组并确认退出后标记取消。
- ZIP 只允许成功任务生成。每个任务使用独立生成锁，先写临时文件再原子替换最终 ZIP；生成失败只影响下载请求，不改变已完成任务状态。

## 部署

应用仍由 Uvicorn 启动，生产运行绑定 `0.0.0.0`，默认端口 `8000`，访问地址为 `http://172.20.0.160:8000`。提供 systemd unit 示例，使用服务器上已经安装 PaddleOCR GPU 依赖的 Python 环境，并配置工作目录、单实例、`--workers 1`、`KillMode=control-group`、`TimeoutStopSec=45`、自动重启和必要环境变量。

服务不内置 HTTPS 或登录。部署说明必须明确要求只开放给可信内网，并通过主机防火墙限制端口来源范围。

## 测试策略

- 任务存储：状态转换、重启恢复、敏感字段不落库。
- GPU 清单：解析正常输出、命令缺失、无 GPU、无效 GPU 编号。
- 调度器：同卡严格串行、异卡可并行、取消排队/运行任务、子进程失败、密钥清理。
- API：合法/非法上传、VLM 必填校验、任务查询、取消、路径穿越防护、单文件下载和 ZIP。
- 页面：静态页面可加载，关键表单字段和 API 路径存在。
- 回归：现有结果浏览 API 和 CLI 测试继续通过。

子进程测试使用可注入的命令执行器或短生命周期假进程，不启动真实 PaddleOCR，也不依赖 GPU 或外部 VLM 服务。

## 完成标准

- 从内网浏览器访问服务，上传一个 PDF 并选择物理 GPU 后能创建任务。
- 页面填写的 VLM Endpoint、模型和 Key 被该任务正确使用，Key 不出现在数据库、API 响应、日志、命令行和 ZIP 中。
- 同卡任务不会同时运行，不同卡任务能够同时运行。
- 页面可观察任务状态和阶段进度，能够取消任务并看到明确错误。
- 成功任务可浏览结果、下载任一输出文件并下载完整 ZIP。
- 服务可以通过 systemd 示例在 GPU 服务器上绑定 `0.0.0.0:8000` 运行。
